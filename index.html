八种常见的数据结构简介

2023-12-13

数据结构是计算机存储、组织数据的方式。选择合适的数据结构可以带来更高的运行效率或存储效率。数据在内存中呈线性排列，但是我们可以使用指针等方式，构造出类似“树形”的复杂结构。下面将依次介绍八种常见的数据结构。

一、数组

数组（Array）是有序元素的序列，在内存中的空间是连续的，数组会为其中的每个元素分配一个下标，下标从0开始，自增且连续。

优点：访问数据简单
缺点：添加和删除数据比较耗时
使用场景：对存储空间要求不大，查询较多，添加和删除较少

1. 数据访问

由于数据是存储在连续空间内，所以每个数据的内存地址都是通过数据下标算出，可以直接访问目标数据。（称做“随机访问”）。如下图，可以直接使用a[2]访问Red。

二、链表

链表（Linked List）是由一系列节点Node组成，数据元素的逻辑顺序是通过链表的指针地址实现，通常情况下，链表中的每个节点包含两个部分，一个用于存储元素的内存地址，名叫数据域；另一个则指向下一个相邻节点的地址，名叫指针域。根据链表的指向不同可分为单向链表、双向链表、循环链表等。下面将介绍单向链表。

优点：数据添加和删除方便
缺点：访问比较耗费时间
适用场景：数据量较小，访问较少，添加和删除较多

1. 数据访问

与数组不同，链表中的元素在内存中的空间不一定是连续的，想要访问某个元素，只能从第一个元素开始，顺着指针的指向进行遍历

三、栈

栈（Stack）是一种特殊的线性表，仅能在线性表的一端操作，栈顶允许操作，栈底不允许操作。 栈的特点是先进后出。从栈顶放入元素的操作叫入栈（压栈），取出元素叫出栈（弹栈）

四、队列

队列（Queue）也是一种线性表，其限制是仅允许在表的一端进行插入，在表的另一端进行删除。队列的特点是先进先出，放入元素的操作叫入队，取出元素叫出队。

五、散列表

散列表（Hash）也叫哈希表，是根据键和值 (key和value) 进行访问的数据结构，通过key和value来映射到集合中的一个位置，这样就可以很快找到对应元素。利用了数组支持按照下标访问的特性，我们也可以认为散列表是数组的一种扩展。

把key通过一个函数（哈希函数）转换为一个整型数值，再对数组长度求余，其结果作为数组下标，将value存储在以该数字为下标的数组中，利用了数组在查找方面的优势，数据获取相对较快。

把key当作数据的标识符，值即为数据的内容。如下图，key为人名，value为性别

如果数组的长度太短，使用哈希表时容易发生冲突，有较大概率需要进行线性查找；如果数组的空间太大，就会出现很多空箱子，造成内存的浪费。因此，给数组设定合适的空间非常重要。在存储数据的过程中，如果发生冲突，可以利用链表在已有数据的后面插入新数据来解决冲突。这种方法被称为链地址法，另外还有开放地址法等解决冲突的方法。

六、树

树（Tree）是一种数据结构，它是由n（n>=1）个有限节点组成一个具有层次关系的集合。树根是朝上的，叶子朝下。它具有以下的特点：

每个节点有0个或多个子节点
没有父节点的节点称为根节点
每一个非根节点有且只有一个父节点
除了根节点外，每个子节点可以分为多个不相交的子树

日常应用中，二叉树使用的频率较高，其特点如下：

每个结点最多有两个子结点
左子树和右子树是有顺序的，次序不能颠倒
即使某结点只有一个子树，也要区分左右子树

二叉树添加，删除元素都很快，并且在查找方面也有很多的算法优化。所以，二叉树既有链表的好处，又有数组的好处，是两者的优化方案，在处理大批量的动态数据方面非常有用。

二叉树有很多扩展的数据结构，包括平衡二叉树、红黑树、B+树等，这些数据结构在二叉树的基础上衍生了很多的功能，在实际应用中广泛用到，例如mysql的数据库索引结构用的就是B+树，还有HashMap的底层源码中用到了红黑树（JDK8以上）。

七、堆

堆（Heap）可以看做是用数组实现的二叉树，所以它没有使用父指针或者子指针。堆根据“堆属性”来排序,“堆属性”决定了树中节点的位置。其特点如下：

每个节点最多有两个子节点
排列顺序必须从上到下，同一行从左到右
堆中某个节点的值总是不大于或不小于其父节点的值
存放数据时，一般会把新数据放在最下面一行靠左的位置。如果最下面一行没有多余空间时，就再往下另起一行，并把数据添加到这一行的最左端

1. 堆的性质

  • 堆是一个完全二叉树（从根结点到倒数第二层满足满二叉树，最后一层可以不完全填充，其叶子结点都靠左对齐）
  • 堆中某个结点的值总是不大于或不小于其父结点的值

将根结点最大的堆叫做最大堆或大根堆，根结点最小的堆叫做最小堆或小根堆。常见的堆有二叉堆、斐波那契堆等
一棵层数为k的有n个结点的二叉树，树中的结点按从上至下、从左到右的顺序进行编号，如果编号为i（1≤i≤n）的结点与满二叉树中编号为i的结点在二叉树中的位置相同，则这棵二叉树称为完全二叉树
一棵层数为k 且有 2^k−1 个结点的二叉树称为满二叉树。也就是说除了叶子节点都有2个子节点，且所有的叶子节点都在同一层

2. 堆和普通树的区别

堆并不能取代二叉搜索树，它们之间有相似之处也有一些不同。主要差别为：

  1. 节点的顺序。在二叉搜索树中，左子节点必须比父节点小，右子节点必须比父节点大。但是在堆中并非如此。在最大堆中两个子节点都必须比父节点小，而在最小堆中，它们都必须比父节点大
  2. 内存占用。普通树占用的内存空间比它们存储的数据要多，要为节点对象以及左右子节点指针分配内存。堆仅仅使用一个数组来存储数据，且不使用指针
  3. 平衡。二叉搜索树必须是“平衡”的情况下，其大部分操作的复杂度才能达到O(log n)。可以按任意顺序位置插入、删除数据，或者使用 AVL 树或者红黑树，但是在堆中实际上不需要整棵树都是有序的。我们只需要满足堆属性即可，所以在堆中平衡不是问题。因为堆中数据的组织方式可以保证O(log n) 的性能
  4. 搜索。在二叉树中搜索较快，但是在堆中搜索较慢。使用堆的目的是将最大（或者最小）的节点放在最前面，从而高效地执行插入、删除操作

八、图

图（Graph）是一系列顶点（元素）的集合，这些顶点通过边连接起来组成图，边是这些顶点之间的连线，图大致可分为无向图和有向图。

下文中将使用图来进一步说明。

  1. 程序启动，完成了类加载后，栈内存和堆内存

    主线程栈中的ThreadLocal的引用指向堆中的ThreadLocal对象(也可称为ThreadLocal实例)，值得注意的是，此时这个ThreadLocal对象并没有值，因为实例化时并未设置初始值

  2. 在main方法中执行以下代码，在主线程中为threadLocalString赋值

    ThreadLocalDemo.threadLocalString.set(“hello world main”);

    在这一步中，主线程的ThreadLocalMap（详见Thread类的threadLocals变量）进行初始化，ThreadLocalMap中有一键值对，此键值对的key指向ThreadLocal对象，即第1步中创建的ThreadLocal对象，不过这个引用是弱引用，弱引用的特点是，进行GC时，无论内存是否足够，仅被弱引用指向的对象一定会被回收

  3. 在main方法中创建新线程，在新线程中为threadLocalString赋值

    ThreadLocalDemo.threadLocalString.set(“new thread”);

    主线程中创建了新线程，在新线程中为threadLocalString赋值，赋值时，初始化了新线程的ThreadLocalMap，该map中同样持有一键值对，此键值对的key指向ThreadLocal对象（弱引用）。通过DEBUG得知，在主线程和新线程中，它们的ThreadLocalMap中，threadLocalString的key确实是同一个对象

  4. 如果主线程和新线程的ThreadLocalMap中key都以强引用的方式指向ThreadLocal实例。我们主动释放两个线程对于ThreadLocal对象的引用

    两个线程都主动释放掉了对ThreadLocal对象的引用，但是，从主线程thread引用->ThreadLocal对象，依然存在这一条可达路径。JVM判断一个对象是否可回收的算法通常为可达路径算法。可达路径算法以GCROOT出发，如果存在一条通向某个对象的强引用通路，那么这个对象是永远不会回收的。thread的引用是主线程的一个本地变量，根据GCROOT算法，thread的引用是可以作为一个GCROOT的。我们显式地释放掉了threadLocal的引用(ThreadLocalDemo.threadLocal = null;)，因为我们确认后续我们不会使用到它了，但是，由于存在GCROOT的一条可达通路，程序并没有像我们希望的那样立刻释放掉ThreadLocal对象，直到我们所有的线程都释放掉了，即程序结束，ThreadLocal对象才会被真正的释放掉，这无疑是内存泄露。

  5. 若使用弱引用

    如上图所示，由于ThreadLocal对象仅被弱引用指向，所以在下一次GC的时候，ThreadLocal对象会被回收。ThreadLocal对象被回收后，主线程和新线程的本例提到的Entry key值变为空，但这个value不能再被访问了，只有当线程执行结束后，对应的value才会被回收。在调用ThreadLocal的get(), set(), remove() 方法的时候都会清除线程ThreadLocalMap里所有key为null的value，减少内存泄漏情况的发生。

ThreadLocal造成内存泄漏

内存泄漏的定义：程序中动态分配的堆内存由于某种原因未释放或无法释放，造成系统内存的浪费，导致程序运行速度减慢甚至系统崩溃等严重后果。

根据上个章节末尾的描述，ThreadLocalMap中，其键值对的key以弱引用指向ThreadLocal对象，如果一个ThreadLocal对象没有强引用指向它，在GC时，ThreadLocal对象会被回收，此时键值对的key值为null。如果当前线程迟迟不结束或不再修改ThreadLocalMap（不修改ThreadLocalMap就无法触发清除无用键值对的机制），这些key为null的键值对一直存在一条强引用链：Thread引用 -> Thread对象 -> ThreadLocalMap -> Entry -> value 永远无法回收，造成内存泄漏。

总得来说，ThreadLocalMap内存泄漏的根本原因是，ThreadLocalMap的生命周期与Thread一样长，如果在使用过后未手动删除对应的key，就有可能出现内存泄漏。

ThreadLocal最佳实践

实际应用中我们会用线程池去维护线程，通常会设置一些核心线程，且不自动销毁，这时需要注意内存泄漏的问题。expungeStaleEntry、cleanSomeSlots、replaceStaleEntry、set、get、remove这些方法都对key为null的entry进行了处理，但这只是官方提供的兜底方案，不应该过度依赖此机制。在使用中注意以下几点：

  1. 使用完成之后，手动remove
  2. 初始化避免空指针问题：ThreadLocal.withInitial(()-> 初始化值);
  3. 把ThreadLocal变量修饰为static

Java的四种引用类型

2023-12-04

Java有强、软、弱、虚四种引用，本文将依次介绍这4种引用。

简介

  1. 强引用(Strong Reference)

    Java中引用默认就是强引用，任何一个对象的赋值操作就产生了这个对象的强引用。强引用的特点是只要有强引用存在，被引用的对象就不会被回收

  2. 软引用(Soft Reference)

    软引用的类定义

    public class SoftReference extends Reference

    SoftReference有两种构造方法

    public SoftReference(T referent)

    public SoftReference(T referent, ReferenceQueue<? super T> q)

    软引用的特点是只有在内存不足的情况下，被引用的对象才会被回收

  3. 弱引用(Weak Reference)

    弱引用与软引用的构造方法类似，同样拥有一个参数的构造方法和两个参数的构造方法。弱引用的特点是只要垃圾回收机制执行，被弱引用的对象就会被回收，不论内存是否充足

  4. 虚引用(Phantom Reference)

    虚引用有且只有以下一个构造方法

    public PhantomReference(T referent, ReferenceQueue<? super T> q)

    虚引用必须和引用队列联合使用，“虚”顾名思义“形同虚设”，与其他几种引用不同，虚引用不会决定对象的生命周期，如果一个对象仅持有虚引用，那就和没有引用一样，在任何时候都可能被回收。通过虚引用不能获取到指向对象的实例。

    虚引用的主要作用是跟踪对象被垃圾回收的状态，在垃圾回收的过程中，如果发现还有虚引用指向待回收的内存区域时，会在回收内存之前，将此虚引用加入到引用队列中，程序可以通过判断引用队列中是否加入了虚引用（调用poll方法），来了解指向区域的内存是否将要被回收

2023-04-02

Redis提供了2种持久化的方式，分别为RDB快照和AOF，本文介绍这2种持久化方式的基本用法和原理。

一、RDB快照

1. 什么是RDB快照

在默认情况下， Redis 将内存数据库快照保存在名字为 dump.rdb 的二进制文件中。可以在redis的配置文件中进行设置， 让它在“ N 秒内数据集至少有 M 个改动”这一条件被满足时，自动保存一次数据。例如，以下命令表示60 秒内有至少有 1000 个键被改动时，自动保存一次。

配置中关于RDB的部分说明如图：

默认情况下当满足以下三个条件之一时，将触发写入操作：

  1. 每3600秒大于等于1个key的值有变化
  2. 每300秒大于等于100个key的值有变化
  3. 每60秒大于等于10000个key的值有变化

如果想关闭快照功能，可以注释掉配置文件中 save <second> <changes>的配置或将其配置为save ""即可。

2. 如何手动操作

第一小节中介绍了如何配置快照功能，其实Redis也为我们提供了2个命令，方便使用者手动生成快照文件。分别为save和bgsave，需要注意的是，此处的save是Redis命令（常见的Redis命令有GET、SET等），需在Redis客户端中执行。而第一小节中提到的save是配置文件中的关键字。save和bgsave区别如下：

bgsave 子进程是由主线程 fork 生成的，可与主线程共享主线程内存中的数据。bgsave 子进程运行后，开始读取主线程的内存数据，并把它们写入 RDB 文件。此时，如果主线程对这些数据做了读操作，那么，主线程和 bgsave 子进程相互不影响。如果主线程要修改数据，这个数据就会被复制一份，生成该数据的副本。然后，bgsave 子进程会把这个副本数据写入 RDB 文件，而在这个过程中，主线程仍然可以直接修改原来的数据。第一小节提到的自动生成RDB文件使用bgsave这种方式。

三、Redis 4.0混合持久化

在Redis启动时如果既有rdb文件，又有aof文件，优先选择aof文件恢复数据，因为理论上aof文件中的数据更全。但如果aof文件比较大，启动时需要花费很长时间，为了解决这个问题，Redis提供了混合持久化的功能。

如果开启了混合持久化后，进行AOF重写时会将重写开始这一刻之前的数据做RDB快照处理，将RDB快照内容和增量的AOF修改内存数据的命令保存在一起，写入新的文件，AOF重写完成后覆盖原来的aof文件。在Redis重启时，先加载文件头部的RDB的内容，再加载文件尾部的AOF日志，使得Redis启动效率大幅提升。

为什么局部内部类和匿名内部类只能访问局部final变量？

之前被这个问题困扰过，直到最近看了几篇文章之后，对Java内部类才有了更进一步的认识。

默认情况下，编译器会为匿名内部类和局部内部类起名为外部类名x.class(x为正整数)，上面这段代码会被编译成2个class文件，Test.class和Test1.class，Test1.class对应test方法中Thread的匿名内部类

在run方法中有一条指令：

bipush 10

这条指令的操作是将操作数10压栈，表示使用的是一个本地局部变量。这个步骤是在编译期间由编译器默认执行，如果变量的值在编译期间可以确定，则编译器默认会在匿名内部类（局部内部类）的常量池中添加一个内容相等的量或直接将相应的字节码嵌入到执行字节码中。这样一来，匿名内部类使用的变量是另一个局部变量，只不过值和方法中局部变量的值相等，得以和方法中的局部变量完全独立开

但是这种方式引入了新的问题，既然在run方法中访问的变量a和test方法中的变量a不是同一个变量，在run方法中改变a的值，会造成数据不一致，为了解决这个问题，java编译器就限定必须将变量a限制为final变量，不允许对变量a进行更改（对于引用类型的变量，是不允许指向新的对象），这样数据不一致性的问题就得以解决了。

GitHub SSH配置

2022-08-21

当前电脑上已经有了公司的GitLab配置，想要再配置GitHub的ssh，操作步骤如下：

1. 创建SSH key，执行命令一路回车即可

ssh-keygen -t rsa -C "注册邮箱" -f ~/.ssh/id_rsa_github

2. 配置config文件

vim ~/.ssh/config

在文件中添加以下内容

HOST github.com
hostname github.com
User github用户名
IdentityFile C:\Users\Administrator\.ssh\id_rsa_github
PreferredAuthentications publickey

3. 将第一步中生成的公钥文件（id_rsa_github.pub）中的内容配置到github中

4. 测试连接

ssh -vT git@github.com

出现以下提示说明连接已成功：
You’ve successfully authenticated, but GitHub does not provide shell access

2. 回滚本地提交记录

命令 影响暂存区 影响工作区
git reset –soft N N 执行此命令后，将当前分支的本地仓库回退到commId对应的那次提交，并将已提交的文件保存到暂存区。工作区和暂存区中的其他文件不受影响
git reset –mixed Y N 执行此命令后，将当前分支的本地仓库回退到commId对应的那次提交，将已提交的文件保存到工作区，暂存区中的其他文件也将被保存到工作区，即清空了暂存区。–mixed是默认参数，如果git reset 命令不加参数，将使用默认参数–mixed
git reset –hard Y Y 执行此命令后，将当前分支的本地仓库回退到commId对应的那次提交，已提交的文件会丢失，原暂存区和工作区会被清空

执行上述命令时，除了使用<commitId>以外，也可以使用head
